What is a Transaction?

Definition

A transaction is a logical unit of work that contains one or more SQL statements.

Key Concept

All operations must complete successfully, or none should take effect.

Example: Bank Transfer Transaction

BEGIN TRANSACTION;
UPDATE accounts SET balance = balance - 100 WHERE account_id = 1;
UPDATE accounts SET balance = balance + 100 WHERE account_id = 2;
COMMIT;

Transaction States

Five States of a Transaction:

  • Active - Initial state, transaction is executing
  • Partially Committed - After final statement executes
  • Committed - After successful completion
  • Failed - After discovery that normal execution can't proceed
  • Aborted - After rollback and database restoration
üí°Transitions: Active ‚Üí Partially Committed ‚Üí Committed OR Active ‚Üí Failed ‚Üí Aborted

ACID Properties

ACID ensures reliable transaction processing:

  • Atomicity - All or Nothing
  • Consistency - Valid State to Valid State
  • Isolation - Transactions Don't Interfere
  • Durability - Changes Persist Forever

These properties guarantee database reliability even in the face of errors, power failures, and other mishaps.

Atomicity

"All or Nothing"

Definition

A transaction must be treated as a single, indivisible unit.

Either:

  • All operations complete successfully (COMMIT)
  • All operations are undone (ROLLBACK)

No partial execution is allowed.

‚öõÔ∏èThink of it like a chemical reaction - either all reactants form products, or nothing changes.

Atomicity - Success Case

Transfer $500 from Alice to Bob

BEGIN TRANSACTION;
UPDATE accounts SET balance = balance - 500 
WHERE account_id = 1;  -- Alice: $5000 ‚Üí $4500

UPDATE accounts SET balance = balance + 500 
WHERE account_id = 2;  -- Bob: $3000 ‚Üí $3500

COMMIT;

Result:

AccountBeforeAfter
Alice$5,000$4,500
Bob$3,000$3,500
‚úÖBoth updates complete - transaction commits successfully

Atomicity - Failure Case

Failed Transfer Scenario

BEGIN TRANSACTION;
UPDATE accounts SET balance = balance - 500 
WHERE account_id = 1;  -- Alice: $5000 ‚Üí $4500

-- System crash / Network error / Constraint violation
-- Before second update completes

ROLLBACK;  -- All changes undone

Result:

AccountAttemptedFinal State
Alice$4,500 (temp)$5,000 (restored)
BobNo change$3,000 (unchanged)
‚ùåTransaction rolled back - database restored to original state

Atomicity - Real Example

E-commerce Order Processing

-- 2. Create order

üí°If ANY step fails, ALL steps must be undone

Consistency

Maintaining Database Rules

Definition

A transaction must take the database from one valid state to another valid state.

Ensures:

  • All integrity constraints are satisfied
  • Business rules are maintained
  • Database remains in a correct state
‚öñÔ∏èExample: Total money in bank = Sum of all balances (before and after)

Consistency Example

Using Our Accounts Table

-- Before transaction
SELECT SUM(balance) FROM accounts;  
-- Returns: $17,500

BEGIN TRANSACTION;
UPDATE accounts SET balance = balance - 1000 WHERE account_id = 1;
UPDATE accounts SET balance = balance + 1000 WHERE account_id = 2;
COMMIT;

-- After transaction
SELECT SUM(balance) FROM accounts;  
-- Still Returns: $17,500
Consistency Rule: Money is neither created nor destroyed
Before: $5,000 + $3,000 + $7,500 + $2,000 = $17,500
After: $4,000 + $4,000 + $7,500 + $2,000 = $17,500 ‚úì

Consistency - Constraints

CHECK Constraint Example

ALTER TABLE accounts 
ADD CONSTRAINT balance_check CHECK (balance >= 0);

-- This transaction will FAIL
BEGIN TRANSACTION;
UPDATE accounts SET balance = balance - 6000 
WHERE account_id = 1;  
-- Alice only has $5,000!
-- Error: CHECK constraint failed: balance >= 0

ROLLBACK;
üö´Database prevents inconsistent state - transaction rejected

Consistency - Foreign Keys

Isolation

Concurrent Execution

Definition

Multiple transactions executing concurrently should not interfere with each other.

The Challenge

Alice and Bob both try to withdraw from the same account simultaneously

Without Isolation: Both see balance = $5000, both withdraw $4000
Result: Account overdrawn by $3000!
With Isolation: One transaction waits, sees updated balance

Lost Update Problem

Using Alice's Account

T1: Read balance = $5,000 (Alice's withdrawal)
T2: Read balance = $5,000 (Bob's deposit)
T1: Write balance = $5,000 - $500 = $4,500
T2: Write balance = $5,000 + $300 = $5,300
‚ö†Ô∏èProblem: T1's withdrawal lost! Should be $4,800, but it's $5,300

Isolation Levels

Four Standard Levels:

  • READ UNCOMMITTED - Lowest isolation, allows dirty reads
  • READ COMMITTED - Prevents dirty reads
  • REPEATABLE READ - Prevents dirty and non-repeatable reads
  • SERIALIZABLE - Highest isolation, prevents all anomalies
üîíTrade-off: Higher isolation = More consistency, Lower performance

Dirty Read Problem

Using Alice's Account

Time T1 (Withdrawal) T2 (Balance Check) 1 BEGIN TRANSACTION 2 UPDATE accounts SET balance = 4500 WHERE account_id = 1 -- Not committed! 3 BEGIN TRANSACTION 4 SELECT balance FROM accounts WHERE account_id = 1 ‚Üí Reads $4500 ‚ùå 5 ROLLBACK -- Balance back to $5000 6 -- T2 has wrong data!
üíÄT2 read uncommitted "dirty" data

Non-Repeatable Read

Using Bob's Account

Time T1 (Report Generation) T2 (Deposit) 1 BEGIN TRANSACTION 2 SELECT balance FROM accounts WHERE account_id = 2 ‚Üí Returns $3000 3 BEGIN TRANSACTION 4 UPDATE accounts SET balance = 3500 WHERE account_id = 2 5 COMMIT 6 SELECT balance FROM accounts WHERE account_id = 2 ‚Üí Returns $3500 ‚ùå
üîÑSame query, different result in same transaction

Phantom Read Problem

Using Our Accounts Table

Time T1 (Report) T2 (New Account) 1 BEGIN TRANSACTION 2 SELECT COUNT(*) FROM accounts WHERE balance > 5000 ‚Üí Returns 2 (Charlie, Alice) 3 BEGIN TRANSACTION 4 INSERT INTO accounts VALUES (5, 'Eve', 6000, 'Checking') 5 COMMIT 6 SELECT COUNT(*) FROM accounts WHERE balance > 5000 ‚Üí Returns 3 ‚ùå
üëª"Phantom" row appeared during transaction

Isolation Levels Comparison

Level Dirty Read Non-Repeatable Phantom
READ UNCOMMITTED ‚ùå Allowed ‚ùå Allowed ‚ùå Allowed
READ COMMITTED ‚úÖ Prevented ‚ùå Allowed ‚ùå Allowed
REPEATABLE READ ‚úÖ Prevented ‚úÖ Prevented ‚ùå Allowed

Setting Isolation Levels

-- PostgreSQL
SET TRANSACTION ISOLATION LEVEL READ COMMITTED;

-- MySQL
SET SESSION TRANSACTION ISOLATION LEVEL REPEATABLE READ;

-- SQL Server
SET TRANSACTION ISOLATION LEVEL SERIALIZABLE;

-- Example with our accounts
SET TRANSACTION ISOLATION LEVEL SERIALIZABLE;
BEGIN TRANSACTION;
SELECT * FROM accounts WHERE account_id = 1;
-- No other transaction can modify this row until we commit
COMMIT;

Durability

Permanent Changes

Definition

Once a transaction commits, its changes must persist even in case of system failures.

Guarantees:

  • Committed transactions survive crashes
  • Changes written to non-volatile storage
  • Recovery mechanisms restore data
Scenario: Alice transfers $1,000, sees "Success", then power failure.
Durability: When system restarts, transfer is still recorded!

Durability - Write-Ahead Log

How It Works

BEGIN TRANSACTION;
UPDATE accounts SET balance = 4000 WHERE account_id = 1;
COMMIT;

-- Behind the scenes:
-- 1. Write to transaction log (disk):
--    "UPDATE accounts SET balance=4000 WHERE account_id=1"
-- 2. Mark log entry as COMMITTED
-- 3. Return success to user
-- 4. Later: Apply changes to actual data files
-- 5. Eventually: Remove from log
üìùEven if crash after COMMIT, log ensures recovery

Concurrency Control

Why We Need It

Multiple users accessing our accounts table simultaneously

Benefits:

  • Higher throughput (more transactions/second)
  • Better response time
  • Efficient resource use

Example Scenario:

100 customers all checking balances, making deposits, transfers at the same time. Without concurrency control: chaos. With it: smooth operation.

Schedules

Definition

A schedule is the order of operations from concurrent transactions

Example with Alice & Bob:

T1: Transfer $500 from Alice to Bob
T2: Check Alice's balance

Operations:
T1: Read(Alice)    -- $5000
T1: Write(Alice)   -- $4500
T1: Read(Bob)      -- $3000
T1: Write(Bob)     -- $3500
T2: Read(Alice)    -- What value does T2 see?
ü§îAnswer depends on the schedule!

Serial Schedules

No Interleaving

Serial Schedule 1: T1 then T2 T1: Read(Alice) ‚Üí $5000 T1: Write(Alice) ‚Üí $4500 T1: Read(Bob) ‚Üí $3000 T1: Write(Bob) ‚Üí $3500 T2: Read(Alice) ‚Üí $4500 Serial Schedule 2: T2 then T1 T2: Read(Alice) ‚Üí $5000 T1: Read(Alice) ‚Üí $5000 T1: Write(Alice) ‚Üí $4500 T1: Read(Bob) ‚Üí $3000 T1: Write(Bob) ‚Üí $3500
‚úÖAlways correct, but slow

Concurrent Schedules

Interleaved Operations

Concurrent Schedule: T1: Read(Alice) ‚Üí $5000 T2: Read(Alice) ‚Üí $5000 T1: Write(Alice) ‚Üí $4500 T2: Read(Bob) ‚Üí $3000 T1: Read(Bob) ‚Üí $3000 T1: Write(Bob) ‚Üí $3500

Advantages:

  • Better CPU utilization
  • Reduced waiting time
  • Higher throughput

Conflict Serializability

Definition

Schedule is conflict serializable if equivalent to some serial schedule

Operations Conflict When:

  • Different transactions
  • Same data item
  • At least one is WRITE

Conflict Types:

  • Read-Write (RW): T1 reads, T2 writes same item
  • Write-Read (WR): T1 writes, T2 reads same item
  • Write-Write (WW): Both write same item

Conflict Example

Using Alice's Account

T1: Transfer $500 from Alice
T2: Add interest to Alice's account

T1: Read(Alice)   -- $5000
T2: Read(Alice)   -- $5000  ‚Üê No conflict (both reads)
T1: Write(Alice)  -- $4500  ‚Üê Conflicts with T2's Read
T2: Write(Alice)  -- $5250  ‚Üê Conflicts with T1's Write
‚ö†Ô∏èConflicts exist! Order matters for correctness
If T1 then T2: $5000 ‚Üí $4500 ‚Üí $4725
If T2 then T1: $5000 ‚Üí $5250 ‚Üí $4750
Different results!

Precedence Graph

Testing Serializability

Method: Create directed graph to detect conflicts

Rules:

  1. Create node for each transaction
  2. Add edge T1 ‚Üí T2 if T1's operation conflicts with and precedes T2's operation
  3. If cycle exists ‚Üí NOT conflict serializable
  4. If no cycle ‚Üí IS conflict serializable
‚úÖNo cycles = Can be reordered to serial schedule

Precedence Graph Example

Three Concurrent Transactions

T1: Read(Alice), Write(Alice)
T2: Read(Alice), Write(Bob)
T3: Read(Bob), Write(Bob)

Conflicts:
T1 ‚Üí T2 (both access Alice, T1 writes first)
T2 ‚Üí T3 (both access Bob, T2 writes first)
Graph: T1 ‚Üí T2 ‚Üí T3 No cycle! This schedule is conflict serializable. Equivalent to serial order: T1, T2, T3

Non-Serializable Example

Schedule with Cycle

T1: Read(Alice), Write(Bob)
T2: Read(Bob), Write(Alice)

T1: Read(Alice)
T2: Read(Bob)
T1: Write(Bob)    -- T1 ‚Üí T2 (conflict on Bob)
T2: Write(Alice)  -- T2 ‚Üí T1 (conflict on Alice)
Graph: T1 ‚Üí T2 T2 ‚Üí T1 (forms a cycle!)
‚ùåCycle detected! NOT conflict serializable

View Serializability

Less Restrictive Definition

Two schedules are view equivalent if:

  • Same initial reads: If T1 reads initial value in S1, same in S2
  • Same read-from: If T1 reads value written by T2 in S1, same in S2
  • Same final writes: If T1 does final write in S1, same in S2
üí°View serializability ‚äá Conflict serializability

Recoverable Schedules

Definition

Transaction commits only after all transactions whose changes it read have committed

Example with Alice's Account:

T1: Write(Alice) = $4500 T2: Read(Alice) = $4500 -- T2 reads from T1 T1: Commit T2: Commit -- T2 commits AFTER T1
‚úÖThis is recoverable - T2 waits for T1

Non-Recoverable Schedule

Dangerous Scenario

T1: Write(Alice) = $4500 T2: Read(Alice) = $4500 -- T2 reads uncommitted data T2: Commit -- T2 commits first! ‚ùå T1: Rollback -- T1 rolls back
üí•Problem: T2 committed based on data that was rolled back!
Database now in inconsistent state - cannot recover

Cascading Rollbacks

The Domino Effect

T1: Write(Alice) = $4500
T2: Read(Alice) = $4500
T3: Read(Alice) = $4500
T4: Read(Alice) = $4500
T1: Rollback ‚Üí T2 must rollback ‚Üí T3 must rollback ‚Üí T4 must rollback
üìâOne failure cascades through multiple transactions

Cascadeless Schedules

Prevention Strategy

Transaction reads value only after the writing transaction commits

Cascadeless Schedule: T1: Write(Alice) = $4500 T1: Commit -- T1 commits first T2: Read(Alice) = $4500 -- Now T2 can read T2: Commit
üõ°Ô∏èNo cascading rollbacks possible - T2 only reads committed data

Lock-Based Protocols

Concurrency Control Mechanism

Use locks to control access to data items

Two Types of Locks:

  • Shared Lock (S): Multiple transactions can read
  • Exclusive Lock (X): Only one transaction can write

Operations:

  • lock-S(item) - Acquire shared lock
  • lock-X(item) - Acquire exclusive lock
  • unlock(item) - Release lock

Lock Compatibility Matrix

Current Lock Requested: Shared Requested: Exclusive
None ‚úÖ Granted ‚úÖ Granted
Shared ‚úÖ Granted ‚ùå Wait
Exclusive ‚ùå Wait ‚ùå Wait
üîêMultiple readers OK, but writers need exclusive access

Locking Example

Transfer Using Locks

BEGIN TRANSACTION;

lock-X(Alice);              -- Get exclusive lock
READ balance FROM accounts WHERE account_id = 1;
UPDATE accounts SET balance = balance - 500 
WHERE account_id = 1;

lock-X(Bob);                -- Get exclusive lock
READ balance FROM accounts WHERE account_id = 2;
UPDATE accounts SET balance = balance + 500 
WHERE account_id = 2;

COMMIT;
unlock(Alice);              -- Release locks
unlock(Bob);

Two-Phase Locking (2PL)

Protocol Rules

Transaction divided into two phases:

Phase 1: Growing Phase

  • Can acquire locks
  • Cannot release any locks

Phase 2: Shrinking Phase

  • Can release locks
  • Cannot acquire new locks
üìàGuarantees conflict serializability

2PL Example

Visualizing the Phases

Growing Phase: lock-X(Alice) ‚Üë lock-X(Bob) ‚Üë Acquiring locks lock-S(Charlie) ‚Üë --- Lock Point (maximum locks held) --- Shrinking Phase: unlock(Alice) ‚Üì unlock(Charlie) ‚Üì Releasing locks unlock(Bob) ‚Üì

Strict Two-Phase Locking

Enhanced 2PL

Rule: Exclusive locks held until transaction commits or aborts

BEGIN TRANSACTION;
lock-X(Alice);
-- ... operations ...
lock-X(Bob);
-- ... operations ...
COMMIT;               -- Release ALL locks here
unlock(Alice);
unlock(Bob);

Benefits:

  • Prevents cascading rollbacks
  • Ensures recoverability
  • Most widely used in practice

Deadlock Scenario

Circular Wait

Time T1 T2 1 lock-X(Alice) ‚úì 2 lock-X(Bob) ‚úì 3 lock-X(Bob) ‚è≥ -- Waiting for T2... 4 lock-X(Alice) ‚è≥ -- Waiting for T1...
üîÑDeadlock! Both transactions waiting forever

Deadlock Prevention

Strategy 1: Wait-Die

Older transaction waits, younger dies

T1 (timestamp=10) wants lock held by T2 (timestamp=20)
‚Üí T1 waits (older)

T2 (timestamp=20) wants lock held by T1 (timestamp=10)
‚Üí T2 dies/rollback (younger)

Strategy 2: Wound-Wait

Older transaction wounds younger, younger waits

Deadlock Prevention

Strategy 3: Ordered Locking

All transactions acquire locks in same order

-- Rule: Always lock in order: Alice, Bob, Charlie, Diana

-- Both transactions follow same order:
T1: lock-X(Alice)  ‚úì
T1: lock-X(Bob)    ‚úì

T2: lock-X(Alice)  ‚è≥ -- Waits for T1
T2: lock-X(Bob)    -- Will wait
‚úÖNo circular wait possible!

Deadlock Detection

Wait-For Graph

Build graph where edge T1 ‚Üí T2 means T1 waiting for lock held by T2

T1 waiting for lock on Alice (held by T2) T2 waiting for lock on Bob (held by T3) T3 waiting for lock on Charlie (held by T1) Graph: T1 ‚Üí T2 ‚Üí T3 ‚Üí T1 Cycle detected! Deadlock exists.

Resolution:

Abort one transaction in the cycle (victim selection)

Timestamp Ordering

Alternative to Locking

Each transaction gets unique timestamp when it starts

Rules:

  • If T1 wants to read/write item last written by T2
  • If TS(T1) < TS(T2): Abort T1 (too old)
  • If TS(T1) > TS(T2): Allow operation
T1 (TS=100) tries to read Alice
Last write by T2 (TS=150)
‚Üí T1 aborted (reading "future" data)

Summary: Concurrency Control

Main Techniques:

  • Lock-Based: 2PL, Strict 2PL
  • Timestamp-Based: Timestamp ordering
  • Optimistic: Validate at commit time
  • Multi-Version: MVCC (Multiple versions)
üéØChoice depends on: workload type, read/write ratio, conflict frequency

Practical Example Review

Real Transaction on Our Database

-- Alice transfers $1000 to Bob with full ACID
SET TRANSACTION ISOLATION LEVEL SERIALIZABLE;
BEGIN TRANSACTION;

-- Get locks and check balance
SELECT balance FROM accounts 
WHERE account_id = 1 FOR UPDATE;  -- Exclusive lock

-- Verify sufficient funds
IF balance >= 1000 THEN
    UPDATE accounts SET balance = balance - 1000 
    WHERE account_id = 1;
    
    UPDATE accounts SET balance = balance + 1000 
    WHERE account_id = 2;
    
    COMMIT;  -- All ACID properties enforced
ELSE
    ROLLBACK;
END IF;
